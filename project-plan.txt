## PART 1: STACK CHANGES & ARCHITECTURE OVERVIEW

---

### 1.1 Revised Technology Stack

**Mobile:** React Native (Expo managed workflow)
**Backend:** Strapi v4 (Node.js 18)
**Database:** MySQL 8.0 (replaces PostgreSQL)
**AI Service:** Python FastAPI
**Job Queue:** MySQL table-based queue (replaces Redis/Bull — simpler, fewer moving parts)
**Storage:** Cloudflare R2 (S3-compatible, zero egress cost)
**GPU:** RunPod serverless (RTX 4090)
**CDN:** Cloudflare (auto-served from R2)

---

### 1.2 Why MySQL Table Queue Instead of Redis/Bull

A MySQL-based job queue uses a `jobs` table with a `status` column and SELECT FOR UPDATE SKIP LOCKED to claim jobs atomically. This eliminates Redis as a dependency entirely, reducing infrastructure complexity for a 6-day build. The trade-off is slightly lower throughput at very high concurrency — irrelevant at MVP scale. The AI service polls the MySQL jobs table every 3 seconds, claims a job by updating its status to `running`, processes it, then writes results back. Strapi polls the same table to report status to the mobile app.

---

### 1.3 Architecture Rule (Unchanged but Important)

The FastAPI AI service and Strapi are completely separate processes. Strapi writes job rows to MySQL. The FastAPI service reads and processes them. They communicate only through the MySQL jobs table and a webhook call from FastAPI to Strapi when a job completes. They never call each other's internal functions. This separation is what makes the AI-assisted build strategy work — each system can be coded and tested independently.

---

### 1.4 High-Level Data Flow

1. User captures 4 photos in the app
2. App calls Strapi → creates session row, receives R2 presigned upload URLs
3. App uploads images directly to Cloudflare R2
4. App calls Strapi → confirms uploads → Strapi inserts a job row in MySQL with status = queued
5. FastAPI job worker polls MySQL → claims job → downloads images from R2 → runs MediaPipe + BiSeNet on CPU → sends payload to RunPod
6. RunPod runs FLUX + IP-Adapter + ControlNet → returns generated images as base64
7. FastAPI upscales results → uploads to R2 → updates MySQL generation_results rows → calls Strapi webhook
8. App polls Strapi every 5 seconds → receives CDN URLs as styles complete → renders previews
9. User taps a style → app triggers focus render (3 angles) → same flow repeats for that one style
10. User swipes the 3-angle set → 2.5D viewer interpolates between images

Note on SELECT FOR UPDATE SKIP LOCKED — the job worker query pattern:
```sql
START TRANSACTION;
SELECT * FROM jobs
  WHERE status = 'queued' AND attempt_count < 3
  ORDER BY queued_at ASC
  LIMIT 1
  FOR UPDATE SKIP LOCKED;
UPDATE jobs SET status = 'running', started_at = NOW() WHERE id = ?;
COMMIT;
```
This is safe under concurrent workers with no Redis needed.

## PART 4: NEW FEATURE — CUSTOM HAIRSTYLE PHOTO UPLOAD & EXTRACTION

---

### 4.1 Feature Description

A user can photograph or upload an image of someone else wearing a hairstyle they like. The system extracts the hairstyle from that image using segmentation and style analysis, then applies that extracted hairstyle to the user's own face using the same FLUX + IP-Adapter pipeline.

This is distinct from the catalog — the user supplies the style reference, not the app. The output is treated as a single custom result alongside the catalog results.

---

### 4.2 Style Extraction Pipeline

**Step 1 — Segmentation**
BiSeNet runs on the reference image and extracts class 17 (hair) as an isolated mask. The masked hair region is cropped and saved as a separate R2 asset. This becomes the extracted_mask_key.

**Step 2 — Style Description via Vision Model**
The cropped hair region is sent to a vision-capable model (FLUX redux or a CLIP-based descriptor) to generate a text description of the hairstyle. This becomes the extracted_style_prompt. Example output: "medium length box braids, neatly parted down the center, smooth texture, dark brown color."

**Step 3 — Prompt Injection**
The extracted_style_prompt is used in place of the base_prompt from the hairstyles table. IP-Adapter FaceID still conditions on the user's face. ControlNet still conditions on head pose from the user's source photos. The only difference is the style descriptor comes from the extracted reference rather than the catalog.

**Step 4 — Optional LoRA Matching**
A fuzzy match is run between the extracted_style_prompt and the catalog's base_prompt fields. If similarity is above 0.75, the matching catalog LoRA is applied automatically. If no match, generation runs without a LoRA (still works, just slightly less sharp on style-specific textures).

**Step 5 — Result Delivery**
The result appears in the session's results alongside catalog styles, labeled "Your Custom Style." It is stored in generation_results with hairstyle_id pointing to a dynamically created hairstyles row with source_type = 'user_uploaded'.

---

### 4.3 UI Flow for Custom Upload

The capture screen gets a new button: "Upload a style photo." This opens the device photo library (expo-image-picker). The user selects one image. The app uploads it to R2 via presigned URL and calls the confirm endpoint. A small card appears in the processing screen labeled "Extracting your custom style..." with a progress indicator. Once extraction completes, the custom style card joins the other style cards in the viewer. The user can swipe through it exactly like catalog styles.

## PART 5: HAIRSTYLE TRAINING DATA POOL

---

### 5.1 Architecture Philosophy

The training pool exists now even if LoRA fine-tuning is not scheduled until V2. The reason is that collecting good training data requires time and user behavior you cannot replicate retroactively. Every generated session and every user-uploaded style photo is a potential training asset. The schema captures it all from day one.

---

### 5.2 Data Sources for the Pool

**Source 1 — Admin-Uploaded Samples**
Admin users can upload curated hairstyle photos via the Strapi admin panel. Each photo is tagged with category, style_label, and notes. These are the highest quality samples and go through a BiSeNet extraction pass automatically on upload to produce a clean hair mask.

**Source 2 — User-Uploaded Reference Photos**
When a user uploads a custom style photo (Section 4), with their consent a copy is added to hairstyle_training_samples with approved_for_training = 0 pending admin review. Admin reviews and approves. This is disclosed in the privacy policy and terms.

**Source 3 — High-Rated Generation Results**
When a generation_result receives a user_rating of 5, the system automatically creates a hairstyle_training_samples entry linking to that generated image. These are generated images, not photos — but they are excellent training material for LoRA fine-tuning because they already have clean hair textures and no background clutter.

---

### 5.3 Training Sample Admin Panel

Built inside Strapi admin using the existing content manager. Admin can:
- Filter samples by category, extraction_status, approved_for_training
- View the original image and extracted mask side by side
- Edit style_label and category
- Approve or reject for training
- Export approved samples as a zip (for use in LoRA training scripts)

No custom admin UI is needed for MVP — Strapi's built-in content manager handles this. A custom admin page for the side-by-side image comparison view is a V2 task.

---

### 5.4 Training Pool → LoRA Pipeline (Future)

When enough approved samples exist per style (target: 80-150 per category), a LoRA fine-tuning run is triggered manually using the FLUX LoRA training script on a rented A100 instance. The resulting checkpoint is uploaded to the RunPod Docker image and the lora_checkpoint field in the hairstyles table is updated. No app update required — the checkpoint is baked into the next RunPod image build.

## PART 7: 6-DAY BUILD PLAN WITH AI CODE GENERATION STRATEGY

---

### 7.1 Core Principle: System-First, Integration-Last

The only way to build this in 6 days with AI assistance is to treat every system as a completely isolated unit with a defined input/output contract. You do not write integration code until each unit works independently. AI code generators produce their best output when given a single, well-bounded system with clear inputs, outputs, and no assumed shared state with other systems.

Each system below can be handed to an AI code generator as a standalone prompt. You then test it in isolation, then wire it to adjacent systems.

---

### 7.2 The 8 Independently Codable Systems

**System 1: MySQL Schema + Migrations**
What it is: All CREATE TABLE statements, indexes, seed data for 10 hairstyles, and a migration runner script.
Input to AI: The full schema from Part 2 of this document.
Output expected: A single SQL file with all tables, a seed.sql with 10 hairstyle rows, and a Node.js script that runs them against a MySQL 8 instance.
Dependencies: None. This is the foundation. Do this first, on its own.
Test: Run migrations against a fresh MySQL 8 Docker container, verify all tables exist, query the hairstyles seed data.

**System 2: Strapi Backend (Auth + Sessions + Jobs)**
What it is: Strapi configured for MySQL, custom content types, custom API controllers for sessions, jobs, training samples, and webhook endpoints.
Input to AI: The API design from Part 3, the schema from Part 2, and the instruction that Strapi uses MySQL via the mysql2 provider.
Output expected: A working Strapi project with: all content types created, all custom routes registered, session creation endpoint returning presigned R2 URLs, job insertion logic using MySQL directly (not Redis), webhook endpoints for AI service callbacks.
Dependencies: System 1 must be running (MySQL with tables).
Test: Use a REST client (Bruno or Postman) to: register a user, log in, create a session, get presigned URLs back, confirm uploads, verify job row appears in MySQL.

**System 3: Cloudflare R2 Upload Module**
What it is: A standalone Node.js module (used by Strapi) that generates presigned PUT URLs for R2, and a Python module (used by FastAPI) that uploads files to R2 using boto3.
Input to AI: R2 bucket name, endpoint URL format, required CORS config, the list of file types and size limits.
Output expected: Two files — r2_uploader.js (Node.js, used in Strapi controllers to generate presigned URLs) and r2_uploader.py (Python, used by FastAPI to upload processed results).
Dependencies: Cloudflare R2 bucket must exist and credentials must be available.
Test: Node.js module — call it, get a URL back, use curl to PUT a file to that URL, verify it appears in R2 dashboard. Python module — run a script that uploads a test image to R2, verify CDN URL is accessible.

**System 4: FastAPI Job Worker (The Queue Poller)**
What it is: The FastAPI application with a background worker that polls the MySQL jobs table using SELECT FOR UPDATE SKIP LOCKED, claims jobs, dispatches to the appropriate processing function, handles retries, and calls the Strapi webhook on completion.
Input to AI: The jobs table schema, the SELECT FOR UPDATE SKIP LOCKED query pattern, the job types (preview_batch, focus_render, style_extraction), the webhook URL format, the retry logic (max 3 attempts, exponential backoff).
Output expected: A FastAPI app with: a background asyncio task that polls MySQL every 3 seconds, a dispatcher that routes each job type to a handler function (stubs are fine for now), a webhook caller that POSTs to Strapi on completion or failure.
Dependencies: System 1 (MySQL running), System 3 (R2 credentials).
Test: Insert a row manually into the jobs table with status = queued. Start the FastAPI worker. Watch it claim the job (status changes to running) and call the webhook stub. Verify retry logic by making the handler throw an error three times.

**System 5: RunPod Pipeline (The GPU Handler)**
What it is: The RunPod serverless Docker image with handler.py that accepts a JSON payload, runs FLUX + IP-Adapter FaceID + ControlNet, returns base64-encoded PNG images.
Input to AI: The handler.py template from Appendix B of the original document, the generation parameters (steps=28, guidance_scale=3.5, resolution=768x1024), the IP-Adapter face_strength range, the prompt assembly format.
Output expected: A complete handler.py, a requirements.txt for the Docker image, and a Dockerfile that starts from the RunPod PyTorch base image, downloads model weights, and sets up the handler.
Dependencies: RunPod account with a funded wallet, access to FLUX.1-dev weights.
Test: Build the Docker image locally (CPU-only test without model weights), then push to RunPod, run a test job via the RunPod dashboard with a sample payload, verify base64 PNG comes back.
Note: This system takes the longest to debug because GPU cold starts are slow. Budget a full day for this system alone within your 6-day plan.

**System 6: Face Analysis + Prompt Builder (Python)**
What it is: Two Python modules — face_analysis.py (MediaPipe + BiSeNet) and prompt_builder.py (assembles the full FLUX prompt from face analysis output + hairstyle row data).
Input to AI: The MediaPipe landmark list, the BiSeNet class list (especially classes 1, 13, 17), the Fitzpatrick-to-prompt mapping table, the prompt assembly order from Part 3 of this document.
Output expected: face_analysis.py that takes an image path, returns a dict with head_pose (yaw, pitch, roll), skin_tone_hex, skin_tone_fitzpatrick, hair_mask_path, face_bbox. prompt_builder.py that takes that dict + a hairstyle dict and returns positive_prompt and negative_prompt strings.
Dependencies: None — these are pure Python, no network calls.
Test: Run against 5 test images from your own phone camera. Print the dict output. Verify head_pose values are sensible (front photo should have yaw near 0). Verify skin_tone classification is correct for different people.

**System 7: Style Extraction Module (Python — NEW Feature)**
What it is: A Python module that takes a reference hairstyle image, extracts the hair region using BiSeNet, generates a text description of the extracted style using a vision model, and returns an extracted_style_prompt string plus the path to the isolated hair mask image.
Input to AI: The BiSeNet class map (class 17 = hair), the task description (extract hair region, generate text descriptor), the expected output format.

### 7.4 How to Write Effective AI Code Generation Prompts

**Rule 1 — Give the AI the exact schema.**
Do not say "create a database model for sessions." Paste the full CREATE TABLE statement and say "use this schema exactly." AI will not invent columns you did not ask for.

**Rule 2 — Specify one system per prompt.**
Never ask for System 2 and System 4 in the same prompt. The AI loses coherence on large cross-system prompts. Keep each prompt scoped to one file or one module.

**Rule 3 — Specify the integration contract, not the internals.**
For each system, tell the AI what goes in and what comes out. Example: "This function takes {hairstyle_slug, face_analysis_json} and returns {positive_prompt, negative_prompt} as strings." The AI fills in the internals. You verify the output matches the contract.

**Rule 4 — Always include error handling in the prompt.**
Add "include try/except blocks for all network calls and database operations, log errors with loguru, re-raise with a descriptive message" to every Python prompt and "add error boundaries and loading states to every component" to every React Native prompt.

**Rule 5 — Test before moving to the next system.**
If a system does not pass its checkpoint, do not start the next system. Carrying broken dependencies into dependent systems creates debugging problems that cost more time than the delay.

**Rule 6 — Use AI for refactoring after the fact, not during.**
Write messy working code first. Once a system passes its checkpoint, hand the working code back to AI and say "refactor this for readability and add comments." This is faster than trying to get perfect code on the first pass.
